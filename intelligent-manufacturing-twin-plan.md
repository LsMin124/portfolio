## 1. 한 문장 정의

"기계가 아니라 **의사결정을 시뮬레이션하는 공장** — DES와 물리 시뮬레이션을 결합한 생산·물류 디지털 트윈 위에서, LLM 에이전트가 자연어 질의를 받아 스스로 실험을 설계·실행하고 근거와 신뢰구간을 들고 답하며, 물리 상태 변경은 승인 게이트를 거치는 지능형제조 시스템을 만든다."

## 2. 왜 이 구조인가

- **디지털트윈 직무 직결**: Omniverse/Isaac Sim + USD, 코-시뮬레이션 아키텍처, 정합성 검증 방법론 — 트윈 직군이 요구하는 증거를 전부 실물로 제시.
- **지능형제조 서사**: 트윈을 "보는 대시보드"에서 끝내지 않고, 에이전트가 시뮬레이션을 도구로 사용해 인지→실험→판단→승인→실행의 자율제조 폐루프를 시연.
- **로보틱스 비전과의 접속**: AMR 반송 + ROS2 브리지가 코어. 기초 이론 보유자의 첫 실전 프로젝트로 적정한 깊이이며, Nav2·멀티로봇으로의 확장 경로가 명시적으로 열려 있음.
- **제어 경험의 증명**: 열-냉각 PID 폐루프(§4.2(e))가 코어에 포함 — 물리 모델링·피드백 제어·생산시스템 이론의 기계공학 3종 세트가 전부 코드로 남는다.
- **차별화**: "에이전트가 트윈에서 실험을 돌려 답한다"는 데모는 조회형 챗봇/대시보드 포트폴리오와 결정적으로 구분됨. 기계공학 배경이 실험 설계·검증의 신뢰성 근거로 코드에 박힘.

## 3. 구조: 2제품, 2저장소 (유지)

어느 시점에 멈춰도 완결된 결과물이 남는다는 v1 원칙을 유지한다. Isaac Sim 리스크가 있는 만큼 이 원칙이 더 중요해졌다.

|---|---|---|
| 정체성 | 하이브리드(DES+물리) 생산·물류 디지털 트윈 | 트윈을 소비·실험하는 자연어 관제 에이전트 |
| 타겟 증거 | 디지털트윈, 시뮬레이션, 로봇 통합, 제어 | 지능형제조, AI 에이전트, 의사결정 자동화 |
| 단독 완결성 | 트윈 + 3D 데모 + 검증 리포트로 완결 | 트윈 저장소를 의존성으로 사용 |

## 4. 제품 1 — manufacturing-twin

### 4.1 아키텍처 (하이브리드 코-시뮬레이션)

```
        [SimPy DES 코어] ←─ 오더/완료 이벤트 ─→ [Isaac Sim (물리·3D·AMR)]
          │   라인 로직·통계                         │   ROS2 토픽/브리지
          │   열-냉각 PID 모듈 (ST2)
          ├─→ [what-if 실행기 API]  ← 파라미터 스윕·N회 반복·통계 반환
          ├─→ [InfluxDB] → [Grafana]
          └─→ [OPC UA 서버 (경량)]  ← 트윈 상태 외부 노출 인터페이스
```

역할 분리가 설계의 핵심: **DES는 고속 탐색**(수백 회 반복이 초 단위), **Isaac은 물리 충실도와 3D**(반송의 실제 거동, 데모 영상). 둘은 오더/완료 이벤트만 교환하는 느슨한 결합.

### 4.2 핵심 구성요소

**(a) DES 코어 (SimPy)**
- 스테이션 3~4개 + 버퍼 + 반송 오더 발행 구조의 직렬 라인. 확률적 사이클타임, MTBF/MTTR 고장 모델, 시나리오 스크립트(정상/부하 증가/설비 고장) 재현.
- **what-if 실행기**: 파라미터(AMR 대수, 버퍼 크기, 사이클타임, 수요율 등)를 받아 N회 반복 실행 후 처리량·체류시간·병목 통계와 신뢰구간을 반환하는 API. **에이전트의 핵심 도구가 됨.**

**(b) 물리·3D 계층 (Isaac Sim) — 신규 코어**
- 창고/라인 씬은 prebuilt 자산 최대 활용(직접 모델링 최소화). AMR 1~2대(Nova Carter 계열 기본 자산).
- DES가 발행한 반송 오더 → AMR이 스테이션 간 이동 수행 → 완료 이벤트 회신.
- 주행은 내장 내비게이션/컨트롤러로 시작, ROS2 브리지로 토픽(위치·상태·임무) 노출. Nav2 풀스택은 확장 항목.

**(c) 데이터·인터페이스 계층**
- ROS2 토픽 + DES 이벤트 → InfluxDB → Grafana (라인 처리량, AMR 가동률, 반송 시간 분포, ST2 온도·냉각 출력).
- OPC UA 경량 서버로 트윈 상태 노출 — 실물 설비 연동 지점을 확보했다는 어필 포인트만 유지하고, 정보 모델 풀 설계에 시간을 쓰지 않는다.

**(d) 정합성 검증 (v1 축 유지, 역할 재포장)**
- Little의 법칙(L=λW) 검증, 대기행렬 이론 예측치와 비교.
- AMR 대수·버퍼 크기 파라미터 스윕 → 처리량 민감도, 병목 식별.
- v2에서의 의미: **what-if 결과를 신뢰할 수 있는 근거**. "에이전트의 답이 왜 맞는가?"에 대한 정면 방어.

**(e) 열-냉각 PID 제어 모듈 — 신규 (제어 축, 2026-07-05 개정)**
- **스테이션 1개(ST2) 한정.** 1차 열 모델: 부하에 비례해 발열하고, 냉각계가 열을 방출한다. 이산 시간 적분으로 DES 위에서 구동.
- **이산 PID 제어기**(anti-windup 포함)가 냉각 출력을 조절해 목표 온도를 추종. 게인은 수동 튜닝 절차(P → I → D)를 문서화하며 결정.
- **열화 시나리오**: 방열 효율이 점진 저하 → 제어기가 냉각 출력을 올려 한동안 온도를 방어(제어 노력 상승 추세 = **조기 징후**) → 출력 포화 후 온도 이탈(= 후기 징후). 잘 튜닝된 제어가 열화를 가릴수록 제어 노력 감시가 중요해진다는 서사가 핵심.
- **경보 설계**: 임계값은 임의값이 아니라 정상 운전 구간 통계 기반 **SPC ±3σ 관리한계**. 경보 품질은 §8의 오탐 지표(precision/recall, 오경보 횟수)로 채점 — 리드타임만 보면 "임계값 낮추기" 치팅이 가능하므로 반드시 쌍으로 측정.
- 효과: 코파일럿 진단 도구에 물류(통계) + 설비 열(물리·제어)의 **2축 진단**이 생기고, 피드백 제어 경험이 코드로 증명된다. 착수 전 반나절 독립 실습(학습 커리큘럼 Step 2b)의 코드가 이 모듈의 초안.

### 4.3 산출물
- **3D 데모 영상 (필수)**: 오더 발생 → AMR 반송 → 대시보드 반영의 풀 사이클. 디지털트윈 직무에서 가장 강한 한 방.
- Grafana 대시보드, 검증 리포트(`docs/validation.md` — Little 검증, 민감도, PID 제어 성능·경보 오탐 분석 포함).
- DES·파이프라인 부분은 Docker Compose 원커맨드 (Isaac은 GUI 실행 절차 문서화로 대체).

## 5. 제품 2 — factory-copilot

### 5.1 에이전트 그래프 (LangGraph)

```
사용자 질의 → [플래너: 의도 분석·도구 선택]
                ├─ 시계열 조회 (Text-to-Flux, InfluxDB)
                ├─ 이상 진단 (물류 병목 통계 + ST2 열·제어 노력 추세)  ← 2축 진단
                ├─ what-if 실험 (실험계획 제시 → DES 반복 실행 → 통계 해석)  ★핵심
                ├─ 문서 검색 (운영 매뉴얼 RAG, 선택)
                └─ 반송·설비 제어 (승인 게이트 필수)
              → [합성·검증: 근거 인용·응답 생성] → (부족 시 플래너 루프)
```

### 5.2 설계 원칙
- **승인 게이트 2단**: 물리 상태 변경은 v1과 동일하게 승인 필수. what-if도 대규모 실험 전 실험계획(변수·범위·반복수)을 먼저 제시.
- **근거 강제**: 진단·제안은 도구가 계산한 수치와 신뢰구간만 근거로. 근거 없으면 "데이터 부족" 응답.
- **킬러 데모 시나리오**: "다음 달 수요가 20% 늘면 감당 가능해?" → 에이전트가 AMR 대수·버퍼 스윕 실험 설계 → DES 수백 회 실행 → "현 구성은 처리량 X% 부족, AMR 1대 증차 시 충족(95% CI ±Y)" 리포트 → 승인 시 트윈 구성 변경 적용.
- **진단 데모 시나리오(보조)**: "ST2 요즘 괜찮아?" → 진단 도구가 냉각 출력 추세·SPC 관리한계 대비 온도를 근거로 "방열 효율 저하 의심, 온도 이탈 전 조기 단계" 보고.
- **멀티턴 루프**: 단일 도구로 부족하면 스스로 추가 조회·실험 결정.

### 5.3 평가 (정량화, v1 방법론 유지)
- 질의 30~50개 평가셋 (조회/진단/what-if/제어/거부 혼합).
- 채점: 도구 라우팅 정확도, 쿼리·실험설계 정확성, groundedness, 승인 게이트 준수율.
- (9주차에 여유 시) API vs RTX 5080 로컬 LLM 비교 — 지능형제조의 온프레미스 제약 대응 서사.

### 5.4 산출물
- Streamlit 채팅 UI + 데모 영상 (조회 → 진단 → what-if → 승인 제어 시나리오).
- 평가 리포트 (`docs/evaluation.md`).

## 7. 기술 스택

| 계층 | 기술 | 선택 이유 (면접 답변 겸용) |
|---|---|---|
| 트윈 로직 | Python, SimPy | 고속 반복 실행이 what-if의 전제 — 물리 심으로는 불가능한 탐색 속도 |
| 제어 | 직접 구현 이산 PID + 1차 열 모델 | 폐루프 제어의 최소 완전 증명 — 튜닝 절차·anti-windup·제어 노력 감시까지 서사화 |
| 물리·3D | Isaac Sim (Omniverse/USD) | 디지털트윈 플랫폼 표준으로 부상, prebuilt 자산, RTX 5080 활용. 셋업 시점 버전 호환 확인 |
| 로봇 | ROS2 (Jazzy 계열) | 브리지·토픽 설계의 실전 증명, Nav2 확장 경로 |
| 저장·가시화 | InfluxDB, Grafana, Docker Compose | 시계열 특화, 가시화 코딩 생략 |
| 인터페이스 | OPC UA (asyncua, 경량) | 실물 연동 지점의 최소 증거 |
| 에이전트 | LangGraph (+API/로컬 LLM) | 인터럽트(승인 게이트), 그래프 제어 흐름 |
| UI | Streamlit | 1인 프로젝트 UI 비용 최소화 |

## 8. 실측 지표 체크리스트 (전부 본인 시스템에서 측정)

- [ ] what-if 실험 처리 속도 (시나리오/분), 반복수 대비 신뢰구간 폭 수렴 곡선
- [ ] DES↔Isaac 이벤트 왕복 지연, 이벤트 유실률
- [ ] Little의 법칙 검증 오차율, 병목 스테이션 식별 결과
- [ ] AMR 반송 임무 성공률, 평균 반송 시간(시나리오별)
- [ ] **온도 제어 성능: 정착 시간, 최대 오버슈트, 정상상태 오차 (게인 튜닝 전/후)**
- [ ] **경보 품질: precision / recall, 시나리오당 오경보 횟수, 탐지 리드타임 (임계값 = 정상 구간 SPC ±3σ, 조건 명시) — 리드타임 단독 보고 금지, 반드시 오탐 지표와 쌍으로**
- [ ] 에이전트: 도구 라우팅 정확도, groundedness, 승인 게이트 준수율
- [ ] (선택) API vs 로컬 LLM: 정확도 격차(%p), 응답 지연(s), 최소 가용 모델 크기

주의: 외부 자료의 예시 수치를 절대 차용하지 않는다. 모든 숫자는 측정 조건과 함께 기록한다.

## 10. 면접 방어 포인트 (예상 질문 → 준비된 답)

1. "왜 시뮬레이터를 두 개 쓰나?" → 역할 분리: DES는 초고속 탐색(수백 회/분), 물리 심은 거동 충실도·3D. 실무 디지털트윈의 코-시뮬레이션 패턴을 축소 구현.
2. "시뮬레이션 결과를 신뢰할 수 있나?" → Little의 법칙·대기행렬 검증 + what-if 응답에 신뢰구간 필수 포함.
3. "'지능형'의 실체가 뭔가?" → 에이전트가 실험을 설계·실행·해석해 근거 기반 제안을 하고, 실행은 승인 게이트를 거치는 폐루프.
4. "PLC 없는 트윈이 현실성 있나?" → 조준 계층이 설비 제어가 아닌 의사결정·물류 오케스트레이션 레이어. OPC UA 인터페이스로 기존 설비와의 연동 지점은 확보.
5. "로봇 실무 경험은?" → ROS2 브리지·반송 파이프라인 직접 구현이 그 증명. Nav2·멀티로봇 확장 로드맵 보유.
6. "기계공학 전공이 어디에 쓰였나?" → ① 1차 열 모델링과 PID 폐루프 설계·튜닝, ② 생산시스템 이론(대기행렬·Little) 기반 실험 설계·검증, ③ 물류 민감도 분석. (확장 시 회전기계 PdM 복원.)
7. "경보 임계값과 PID 게인은 어떻게 정했나?" → 임계값은 정상 구간 SPC ±3σ 관리한계, 품질은 precision/recall·오경보 횟수로 측정 (리드타임 단독 보고의 치팅 가능성까지 선제 언급). 게인은 문서화된 수동 튜닝 절차 + 정착시간/오버슈트 실측.
8. "제어가 좋으면 열화가 안 보이지 않나?" → 정확한 지적 — 그래서 온도가 아니라 제어 노력(냉각 출력) 추세를 조기 지표로 감시한다. 폐루프가 이상을 가리는 현상 자체를 진단 설계에 반영했다.
