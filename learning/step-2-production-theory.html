프로젝트와의 연결 — v2 §4.2(d), §5.2

"시뮬레이션인데 그 결과를 왜 믿어야 하죠?"는 이 포트폴리오가 받게 될 가장 확실한 질문이다. 이 스텝의 내용이 그 방어선이고, 동시에 에이전트 what-if 응답에 붙는 신뢰구간의 출처다. 기계공학 전공 증명이 코드로 박히는 지점이기도 하다.

1. Little의 법칙 — L = λW

안정된(정상상태) 시스템에서 시스템 안의 평균 개체 수 L은 도착률 λ와 평균 체류시간 W의 곱과 같다. 분포 가정이 없는, 흐름 보존만으로 성립하는 법칙이라 검증 척도로 이상적이다.

핵심은 검증 방법이다. 세 값을 서로 독립적인 방식으로 측정해서 등식이 맞는지 본다:

  • L — WIP(라인 내 재공품 수)의 시간가중 평균. WIP 변화 이벤트마다 (시각, 수량)을 기록해 면적/시간으로 계산.
  • λ — 관측 구간의 완성품 수 ÷ 구간 길이 (정상상태에서 도착률 = 처리율).
  • W — 부품별 (완성 시각 − 투입 시각)의 평균.
little_check.py — 삼중 독립 측정
import numpy as np

def time_weighted_wip(wip_log, t0, t1):
    """wip_log: 시간순 [(시각, 그 시각부터의 WIP 수)] 리스트"""
    area = 0.0
    pairs = list(zip(wip_log, wip_log[1:] + [(t1, 0)]))
    for (t, w), (t_next, _unused) in pairs:
        area += w * (t_next - t)
    return area / (t1 - t0)

L   = time_weighted_wip(wip_log, t_warmup, t_end)
lam = len(done) / (t_end - t_warmup)
W   = np.mean([fin - start for start, fin in cycle_log])

err = abs(L - lam * W) / L * 100
print(f"L={L:.2f}  λW={lam * W:.2f}  오차율 {err:.2f}%")

오차율이 수 % 이내로 수렴하면(관측 구간을 늘릴수록 줄어야 정상) 시뮬레이터의 흐름 회계가 새지 않는다는 뜻이다. 이 그래프 한 장이 검증 리포트의 첫 페이지가 된다.

2. M/M/1 — 이론값이 있는 과녁

도착이 포아송(율 λ), 서비스가 지수분포(율 μ), 서버 1대, 대기열 무한인 시스템은 닫힌 형태의 해가 존재한다. 이용률 ρ = λ/μ (ρ가 1보다 작아야 안정)일 때:

  • 평균 체류시간 W = 1 / (μ − λ)
  • 시스템 내 평균 개체 수 L = ρ / (1 − ρ)

미니 라인을 일부러 M/M/1 설정으로 단순화(스테이션 1개, 버퍼 무한, 지수 서비스)해서 돌리면 시뮬레이션 결과를 이론값이라는 과녁에 대고 쏠 수 있다. "임의로 만든 시뮬레이터가 아니라 이론 검증을 통과한 엔진"이라는 서사가 여기서 나온다.

3. Warm-up — 빈 라인의 거짓말 걷어내기

시뮬레이션은 텅 빈 라인에서 시작하므로 초반 통계(짧은 체류시간, 낮은 WIP)는 정상상태를 대표하지 못한다. 처리 방법은 실용적으로:

  • WIP의 시계열을 그려 평평해지는 지점을 눈으로 확인하고, 그 앞을 잘라낸다.
  • 관례적 안전값으로 전체 실행의 앞 10~20%를 버리고, 잘라낸 뒤에도 결과가 안정적인지 확인한다.
  • 정석 기법이 궁금하면 Welch's method를 검색 — 리포트에 언급하면 깊이가 보인다.

4. 반복 실행과 신뢰구간 — what-if 응답의 문법

확률 시뮬레이션의 1회 실행은 표본 하나일 뿐이다. 결론은 반드시 서로 다른 시드의 독립 복제 n회 위에서 내린다. 평균의 95% 신뢰구간:

x̄ ± t0.975, n−1 · s / √n

replication.py — what-if 실행기의 심장
import numpy as np
from scipy import stats

def confidence_interval(samples, level=0.95):
    arr = np.asarray(samples, dtype=float)
    n = arr.size
    half = stats.t.ppf((1 + level) / 2, n - 1) * arr.std(ddof=1) / np.sqrt(n)
    return arr.mean(), half

throughputs = [run_once(seed=s).throughput for s in range(20)]
mean, half = confidence_interval(throughputs)
print(f"처리량 {mean:.2f} ± {half:.2f} 개/h (95% CI, n=20)")

프로젝트의 what-if 실행기는 결국 run_once를 파라미터 세트별로 n회씩 돌리고 이 함수를 통과시키는 루프다. 에이전트가 "AMR 3대면 처리량 X ± Y"라고 답할 때의 ± Y가 바로 이것.

5. 병목 식별 — 쌓이는 곳 앞, 굶는 곳 뒤

병목 스테이션의 신호는 두 가지가 함께 나타난다:

  • 가동률이 가장 높다 — 바쁨 시간 ÷ 전체 시간이 1에 가장 가깝다.
  • 앞 버퍼는 만성적으로 차 있고, 뒤 버퍼는 만성적으로 비어 있다 — 상류는 적체, 하류는 기아.

버퍼 크기·사이클타임을 스윕하면서 처리량 변화를 보면 "병목이 아닌 곳을 개선해도 처리량은 늘지 않는다"는 제약이론(TOC)의 교훈이 그래프로 나온다 — 검증 리포트의 두 번째 페이지.

흔한 함정
  • 1회 실행으로 "늘었다/줄었다" 결론 내리기 — 신뢰구간이 겹치면 차이를 주장할 수 없다.
  • 복제마다 같은 시드를 재사용 — 독립성이 깨져 신뢰구간이 무의미해진다.
  • warm-up 미제거 상태로 Little 검증 — 오차가 이유 없이 커 보인다.
  • ρ가 1에 가까운(0.95+) 설정으로 검증 — 수렴이 극도로 느려져 무한정 기다리게 된다. 검증은 ρ 0.7~0.85 권장.
