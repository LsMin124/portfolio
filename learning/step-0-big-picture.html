왜 이 스텝인가

부품을 만들 때 전체 조립도가 머리에 없으면 "지금 이걸 왜 배우지?"에서 동력이 샌다. 이 스텝은 v2 기획서 §4–5의 아키텍처를 학습자 시점으로 다시 그린 조립도다.

핵심 설계 사상 세 가지

1. 시뮬레이터가 두 개인 이유 (코-시뮬레이션)

SimPy는 논리와 속도, Isaac은 물리와 눈. "AMR을 3대로 늘리면?" 같은 질문에 답하려면 시나리오를 수백 번 돌려 통계를 내야 하는데, 물리 시뮬레이터로는 불가능한 속도다. 반대로 반송 로봇이 실제로 움직이는 모습과 물리적 거동은 DES가 못 보여준다. 그래서 둘을 오더/완료 이벤트만 주고받는 느슨한 결합으로 묶는다. 이 역할 분리를 설명할 수 있으면 면접 질문 1번(§10)은 끝난 것이다.

2. what-if가 "지능형"의 실체

에이전트가 단순 조회를 넘어 스스로 실험을 설계하고(변수·범위·반복수), DES에서 실행하고, 신뢰구간과 함께 해석해서 답한다. 트윈의 존재 이유(시뮬레이션 기반 의사결정)를 자연어 인터페이스로 시연하는 것.

3. 물리 상태 변경은 반드시 승인 게이트

읽기(조회·진단·실험)는 자유, 쓰기(반송 오더 투입, 설비 파라미터 변경)는 실행 계획 제시 → 사용자 승인 → 실행. LangGraph의 인터럽트 기능으로 구현한다. 제조 현장 안전 서사의 핵심이다.

반송 오더의 일생 — 데이터 흐름 내러티브

구성요소를 외우지 말고, 오더 하나를 따라가라.

  1. 탄생 — SimPy에서 스테이션 1이 가공을 마치면 "부품을 스테이션 2로 옮겨라"는 반송 오더가 발행된다.
  2. 전달 — 오더는 브리지를 거쳐 ROS2 토픽으로 발행된다 (예: /transport/order).
  3. 수행 — Isaac Sim의 AMR이 오더를 구독해 스테이션 1로 이동, 적재, 스테이션 2로 주행한다.
  4. 보고 — 도착하면 완료 이벤트가 ROS2 → 브리지 → SimPy로 회신되고, DES 라인 로직이 다음 공정을 진행한다.
  5. 기록 — 이 모든 이벤트(발행 시각, 완료 시각, AMR 상태)가 InfluxDB에 시계열로 적재되고 Grafana에 그려진다.
  6. 노출 — 트윈의 현재 상태 요약(처리량, 가동률)은 OPC UA 노드로도 노출된다 — 실물 설비와 연동 가능한 인터페이스의 증거.
  7. 소비 — 사용자가 에이전트에게 "오늘 반송 평균 얼마나 걸렸어?"라고 물으면 InfluxDB를 조회해 근거와 함께 답한다.

용어 지도 — 앞으로 계속 나올 말들

디지털 트윈
실물(또는 가상) 시스템의 상태를 데이터로 미러링하고, 시뮬레이션으로 미래를 실험할 수 있는 가상 대응물.
DES (이산사건 시뮬레이션)
시간이 연속으로 흐르지 않고 "다음 이벤트"로 점프하는 시뮬레이션. 대기·가공·고장 같은 로직에 최적.
코-시뮬레이션
서로 다른 시뮬레이터 두 개가 이벤트/데이터를 교환하며 함께 도는 구조.
what-if 실험
"파라미터를 바꾸면 결과가 어떻게 변하나"를 시뮬레이션 반복 실행으로 답하는 것.
AMR
Autonomous Mobile Robot. 경로를 스스로 계획해 주행하는 반송 로봇.
ROS2 노드/토픽
노드는 프로그램 단위, 토픽은 노드들이 pub/sub으로 메시지를 주고받는 이름 붙은 채널.
USD
Universal Scene Description. Omniverse/Isaac의 3D 씬 포맷이자 합성 체계.
Little의 법칙
L = λW. 시스템 내 평균 개체 수 = 도착률 × 평균 체류시간. 시뮬레이션 정합성 검증의 척도.
MTBF / MTTR
평균 고장 간격 / 평균 수리 시간. 설비 고장 모델의 두 파라미터.
OPC UA
산업 장비의 데이터·제어를 표준화된 주소공간으로 노출하는 프로토콜.
groundedness
LLM 응답의 주장이 도구가 계산한 근거 데이터에 실제로 기반한 정도.
승인 게이트 (HITL)
에이전트가 상태를 바꾸기 전 사람의 승인을 기다리는 중단점. Human-in-the-loop.
