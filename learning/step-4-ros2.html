04

ROS2 기초 — 오더가 흐르는 길 만들기

코딩 ~12h v2 로드맵: 1–2주차 병행 (일 1–2h)

학습 목표

  • ROS2의 통신 모델(노드·토픽·서비스·액션)을 구분하고 용도를 설명할 수 있다.
  • Ubuntu에 ROS2를 설치하고 환경을 잡을 수 있다.
  • rclpy로 퍼블리셔/서브스크라이버 노드 한 쌍을 작성할 수 있다.
  • CLI 도구(ros2 topic, rqt_graph)로 시스템을 관측할 수 있다.
  • "외부 시스템 ↔ ROS2 브리지"의 구조를 코드로 감 잡는다.
프로젝트와의 연결 — v2 §4.1

DES가 발행한 반송 오더는 ROS2 토픽을 타고 Isaac의 AMR에게 가고, 완료 이벤트는 반대로 돌아온다. 이 스텝의 미션(오더 발행 노드 + 가짜 AMR 노드)이 그 브리지의 원형이다 — Step 5에서 가짜 AMR 자리에 진짜 Isaac AMR을 꽂으면 코-시뮬레이션이 완성된다. 로봇 직군 어필의 실체이기도 하다.

1. ROS2는 OS가 아니다

ROS2는 로봇 소프트웨어를 작은 프로그램(노드)들의 통신망으로 짜게 해주는 미들웨어 + 도구 모음 + 생태계다. 통신은 DDS라는 산업용 pub/sub 표준 위에서 돌고, 노드들은 중앙 서버 없이 서로를 자동 발견한다. 핵심 어휘 세 개:

  • 노드(node) — 한 가지 일을 하는 프로그램 단위 (오더 발행기, AMR 컨트롤러).
  • 토픽(topic) — 이름 붙은 방송 채널 (/transport/order). 발행자와 구독자는 서로를 몰라도 된다.
  • 메시지(message) — 토픽에 흐르는 타입 있는 데이터 (std_msgs/msg/String, geometry_msgs/msg/Twist).

2. 통신 4형태 — 언제 무엇을 쓰나

형태패턴전형적 용도이 프로젝트에서
토픽스트림 방송 (N:M)센서값, 상태, 지령 스트림반송 오더, 완료 이벤트, AMR 위치 — 주력
서비스요청 → 응답 (1회)설정 조회, 즉답 계산필요 시 트윈 상태 스냅숏 질의
액션장기 작업 + 진행 피드백 + 취소"X 좌표로 이동하라"Nav2 확장 시 사용 (확장 예약)
파라미터노드별 설정값속도 한계, 토픽 이름브리지 설정

설계 원칙: 사전 학습과 프로젝트 코어는 토픽만으로 간다. 오더/완료를 액션으로 만드는 게 정석에 가깝지만, 토픽 + JSON 페이로드가 학습 마찰이 훨씬 적고 코어 요구를 충족한다. "왜 액션 안 썼나"는 확장 로드맵으로 답하면 된다.

3. 설치와 환경 — 여기서 반나절 잃지 말 것

  1. Ubuntu 24.04 기준 배포판은 Jazzy (LTS). 공식 문서의 apt 설치 절차를 따른다 (데스크톱 풀 설치 권장: ros-jazzy-desktop).
  2. 설치 후 매 셸에서 source /opt/ros/jazzy/setup.bash가 필요하다 — ~/.bashrc에 등록해라.
  3. 동작 확인은 공식 데모: 터미널 A에 talker, B에 listener를 띄워 메시지가 흐르면 성공.
환경 함정 3대장
  • ros2: command not found — 십중팔구 source 누락이다. 새 터미널마다 의심해라.
  • conda/venv와의 충돌 — rclpy는 시스템 파이썬에 묶여 있다. ROS2 작업은 가상환경을 끄고 시스템 파이썬으로 하는 게 초심자 표준. (SimPy 쪽 venv와 섞이면 import 에러 지옥이 열린다.)
  • 같은 네트워크의 다른 ROS2 머신과 토픽이 섞이면 ROS_DOMAIN_ID를 고유값으로 설정.

4. rclpy — 오더 발행자와 가짜 AMR

노드는 Node를 상속한 클래스다. 아래 한 쌍이 미션의 골격이자 브리지의 원형이다:

order_publisher.py
import json
import rclpy
from rclpy.node import Node
from std_msgs.msg import String

class OrderPublisher(Node):
    def __init__(self, orders):
        super().__init__("order_publisher")
        self.pub = self.create_publisher(String, "/transport/order", 10)
        self.orders = list(orders)
        self.timer = self.create_timer(1.0, self.tick)   # 1초마다 하나씩

    def tick(self):
        if not self.orders:
            return
        msg = String()
        msg.data = json.dumps(self.orders.pop(0))
        self.pub.publish(msg)
        self.get_logger().info(f"오더 발행: {msg.data}")

def main():
    rclpy.init()
    orders = [{"id": i, "src": "ST1", "dst": "ST2"} for i in range(5)]
    node = OrderPublisher(orders)
    rclpy.spin(node)          # 콜백 루프 — Ctrl+C로 종료

if __name__ == "__main__":
    main()
amr_mock.py — 3초 뒤 완료를 회신하는 가짜 로봇
import json
import rclpy
from rclpy.node import Node
from std_msgs.msg import String

class AmrMock(Node):
    def __init__(self):
        super().__init__("amr_mock")
        self.sub = self.create_subscription(
            String, "/transport/order", self.on_order, 10)
        self.done_pub = self.create_publisher(String, "/transport/done", 10)

    def on_order(self, msg):
        order = json.loads(msg.data)
        self.get_logger().info(f"오더 {order['id']} 수신 — 반송 시작")
        holder = {}
        def finish():                      # 원샷 타이머 패턴
            holder["t"].cancel()
            done = String()
            done.data = json.dumps({"id": order["id"], "status": "done"})
            self.done_pub.publish(done)
        holder["t"] = self.create_timer(3.0, finish)   # 주행 시간 흉내

def main():
    rclpy.init()
    rclpy.spin(AmrMock())

if __name__ == "__main__":
    main()

콜백 안에서 time.sleep으로 기다리면 노드 전체가 멈춘다 — 그래서 타이머로 완료를 지연 발행한다. 이 "블로킹 금지" 감각은 Isaac 브리지에서도 그대로 필요하다.

5. 관측 도구 — 눈이 있어야 디버깅한다

terminal
ros2 node list                      # 살아있는 노드
ros2 topic list                     # 흐르는 채널
ros2 topic echo /transport/order    # 채널 도청 — 최강의 디버거
ros2 topic hz /transport/done       # 발행 주기 측정
ros2 interface show std_msgs/msg/String
rqt_graph                           # 노드-토픽 연결 그래프 GUI

QoS(서비스 품질)는 지금은 기본값으로 충분하다. 다만 Isaac/Gazebo 브리지에서 토픽이 안 보이면 QoS 불일치(reliability 설정)를 먼저 의심하라 — Step 5에서 다시 만난다.

MISSION 04

브리지 원형: 오더 왕복 루프

  1. 공식 튜토리얼의 talker/listener 데모로 설치를 검증한다.
  2. 위 노드 한 쌍을 ament_python 패키지(imtwin_bridge)로 만들어 colcon build → ros2 run으로 실행한다 (스크립트 직접 실행 말고 정석 경로로).
  3. Step 1 미션에서 쌓은 orders 리스트(JSON 파일로 export)를 order_publisher가 읽어 발행하도록 바꾼다 — DES 산출물이 ROS2 세계로 들어가는 첫 순간.
  4. 세 번째 노드 done_logger를 추가: /transport/done을 구독해 (수신 시각, 오더 id)를 CSV로 기록 — Step 3 파이프라인에 적재할 수 있는 형태로.
  5. rqt_graph로 세 노드의 연결 그래프를 캡처해 연습 저장소에 남긴다.
  6. ros2 topic echo와 hz로 오더 흐름을 관측하고, 발행 주기를 0.2초로 줄였을 때 유실이 없는지 확인한다.
