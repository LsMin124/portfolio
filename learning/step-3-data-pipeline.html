시계열 파이프라인 — InfluxDB, Grafana, Docker Compose

인프라 ~5h v2 §4.2(c) 데이터 계층

학습 목표

  • 시계열 DB의 데이터 모델(measurement / tag / field / timestamp)을 설계 기준과 함께 이해한다.
  • Docker Compose로 InfluxDB + Grafana 스택을 원커맨드 기동한다.
  • 파이썬 클라이언트로 SimPy 이벤트 로그를 적재한다 — 시뮬레이션 시간 매핑 문제 포함.
  • Grafana에서 처리량·WIP 패널을 만든다.
프로젝트와의 연결 — v2 §4.2(c)

트윈의 모든 이벤트(오더 발행·완료, AMR 상태, 스테이션 지표)가 이 파이프라인에 쌓이고, 에이전트의 시계열 조회 도구가 이 DB를 읽는다. 여기서 만드는 compose 파일이 프로젝트 저장소의 docker-compose.yml 초안이다.

1. 왜 시계열 DB인가

트윈 데이터의 질의는 거의 전부 "어느 시간 범위의 어느 대상 지표"다. 시계열 DB는 시간을 1차 축으로 두고 대량 삽입과 시간 범위 스캔에 최적화되어 있고, 보존 정책(오래된 데이터 자동 삭제)과 다운샘플링을 기본기로 제공한다. 범용 RDB로도 되지만, 이 프로젝트에선 "제조 데이터 스택다움"도 선택 이유다.

2. InfluxDB 데이터 모델 — tag와 field의 구분이 전부

한 레코드(포인트)는 measurement + tags + fields + timestamp로 구성된다. 라인 프로토콜로 보면 구조가 명확하다:

line protocol — 합성 예시
line_state,station=ST1 wip=3i,util=0.83 1751360400000000000
transport,amr=amr01,from=ST1,to=ST2 duration_s=42.5 1751360442000000000
─────────  ──────────────  ─────────────────  ───────────────────
measurement      tags            fields        timestamp (ns)
  • tag — 필터·그룹핑 기준 (스테이션명, AMR ID, 시나리오명). 인덱싱되며 항상 문자열.
  • field — 실제 측정값 (WIP, 가동률, 반송 시간). 인덱싱되지 않는다.
  • 판단 기준: WHERE/GROUP BY에 쓸 것인가? → tag. 집계할 수치인가? → field.
  • 고카디널리티 주의 — 부품 ID처럼 값이 무한히 늘어나는 것을 tag로 두면 인덱스가 폭발한다. field로 두거나 저장을 재고하라.
버전 확인 포인트 — 2.x vs 3.x

InfluxDB는 2.x(질의 언어 Flux)와 3.x(SQL 계열) 세대가 갈린다. v2 기획서의 에이전트 조회 도구는 Text-to-Flux를 가정하므로 학습은 2.x 기준으로 시작하되, 프로젝트 착수 시점에 버전·질의 언어를 최종 확정하라 (Text-to-SQL로 바꿔도 도구 구조는 동일하다). 데이터 모델 개념은 두 세대가 같다.

3. Docker Compose — 원커맨드 스택

docker-compose.yml
services:
  influxdb:
    image: influxdb:2.7
    ports:
      - "8086:8086"
    environment:
      DOCKER_INFLUXDB_INIT_MODE: setup
      DOCKER_INFLUXDB_INIT_USERNAME: ${INFLUX_USER}
      DOCKER_INFLUXDB_INIT_PASSWORD: ${INFLUX_PASSWORD}
      DOCKER_INFLUXDB_INIT_ORG: imtwin
      DOCKER_INFLUXDB_INIT_BUCKET: factory
      DOCKER_INFLUXDB_INIT_ADMIN_TOKEN: ${INFLUX_TOKEN}
    volumes:
      - influx-data:/var/lib/influxdb2

  grafana:
    image: grafana/grafana:11.1.0
    ports:
      - "3000:3000"
    volumes:
      - grafana-data:/var/lib/grafana
    depends_on:
      - influxdb

volumes:
  influx-data:
  grafana-data:

비밀값은 같은 폴더의 .env 파일에 두고(INFLUX_USER=... 형식), .env는 반드시 .gitignore에 추가한다. compose 파일에 토큰을 직접 적는 순간 그 저장소는 공개할 수 없게 된다. 이미지 태그는 예시이니 착수 시점의 안정 버전으로 고정하라 (latest 금지 — 재현성).

4. 파이썬 적재 — 시뮬레이션 시간을 어디에 매핑할 것인가

여기가 이 스텝의 실전 포인트다. SimPy의 시각은 0에서 시작하는 논리 시간(분)이고, InfluxDB는 실제 타임스탬프를 요구한다. 관례적 해법은 가상 기준 시각(sim epoch)을 정해 오프셋하는 것:

ingest.py — 시뮬레이션 로그 백필
import os
from datetime import datetime, timedelta, timezone
from influxdb_client import InfluxDBClient, Point
from influxdb_client.client.write_api import SYNCHRONOUS

SIM_EPOCH = datetime(2026, 7, 1, 9, 0, tzinfo=timezone.utc)

def sim_to_ts(sim_minutes):
    return SIM_EPOCH + timedelta(minutes=sim_minutes)

client = InfluxDBClient(
    url="http://localhost:8086",
    token=os.environ["INFLUX_TOKEN"],   # 하드코딩 금지
    org="imtwin",
)
write = client.write_api(write_options=SYNCHRONOUS)

# wip_rows: Step 1 미션에서 쌓은 [(sim_t, station, wip)] 리스트
for sim_t, station, wip in wip_rows:
    point = (
        Point("line_state")
        .tag("station", station)
        .field("wip", int(wip))
        .time(sim_to_ts(sim_t))
    )
    write.write(bucket="factory", record=point)

client.close()

운영 모드는 두 가지가 있고 프로젝트에서 둘 다 쓰게 된다:

  • 백필(batch) — what-if처럼 고속으로 돌린 결과를 한 번에 적재. 위 코드 방식.
  • 실시간 재생(streaming) — 데모용. 시뮬레이션을 벽시계 속도에 맞춰 돌리며(예: SimPy 1분 = 실제 1초) 이벤트 발생 즉시 현재 시각으로 적재.

5. Grafana — 첫 대시보드

  1. localhost:3000 접속 → 초기 비밀번호 변경.
  2. Data source에 InfluxDB 추가 (URL은 컨테이너 네트워크상 http://influxdb:8086, 토큰·org·bucket 입력).
  3. 패널 1: 스테이션별 WIP 시계열 (tag station으로 그룹).
  4. 패널 2: 시간당 처리량 (완성 이벤트 카운트를 1h 윈도우로 집계).
  5. 대시보드를 JSON으로 export해 저장소에 커밋 — 대시보드도 코드다.
흔한 함정
  • 타임스탬프 정밀도 — 클라이언트/DB의 기본 단위(ns)와 직접 만든 정수 타임스탬프가 안 맞으면 1970년이나 먼 미래에 데이터가 찍힌다.
  • Grafana의 기본 시간 창은 "최근 6시간" — SIM_EPOCH를 과거로 잡았으면 시간 범위를 그 근처로 옮겨야 데이터가 보인다.
  • 볼륨 없이 컨테이너를 지우면 데이터가 같이 사라진다.
  • 부품 ID를 tag로 넣는 고카디널리티 실수 (위 §2 참고).
MISSION 03

미니 라인을 대시보드에 띄우기

  1. docker compose up -d 한 번으로 InfluxDB + Grafana가 뜨는 compose 파일을 작성한다 (.env 분리 포함).
  2. Step 1 미니 라인의 이벤트 로그(WIP 변화, 완성, 반송 오더 스텁)를 스키마 설계 후 적재한다 — measurement·tag·field 선택 이유를 주석으로 남긴다.
  3. Grafana에 WIP 패널과 시간당 처리량 패널을 만든다.
  4. 고장 시나리오(ST2 MTTR 2배)를 다시 돌려 적재하고, 대시보드에서 고장 구간이 시각적으로 보이는지 확인한다.
  5. 대시보드 JSON을 export해 연습 저장소에 커밋한다.
  6. 스택 전체를 내렸다 올려도(down → up) 데이터와 대시보드가 살아있는지 확인한다.
