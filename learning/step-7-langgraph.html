LangGraph 에이전트 — 그래프, 도구, 그리고 승인 게이트

코딩 ~10h v2 §5 코파일럿의 뼈대

학습 목표

  • LLM 에이전트의 구성(LLM + 도구 + 제어 흐름)과 각자의 책임을 설명할 수 있다.
  • LangGraph의 상태·노드·엣지·체크포인터 개념을 이해한다.
  • 도구 2개를 가진 미니 에이전트를 만들어 질문이 올바른 도구로 라우팅되는 것을 확인한다.
  • 인터럽트로 승인 게이트(HITL)를 구현하는 패턴을 익힌다.
프로젝트와의 연결 — v2 §5.1–5.2

factory-copilot의 4개 도구(조회/진단/what-if/제어)는 전부 이 스텝의 패턴 위에 선다. 특히 승인 게이트 = LangGraph 인터럽트라는 대응이 v2 설계의 축이고, "LLM이 오작동하면?"이라는 면접 질문의 기술적 답이다.

1. 에이전트의 해부 — 누가 무엇을 책임지나

  • LLM — 의도를 해석하고 "다음에 어떤 도구를 어떤 인자로 부를지"를 결정한다. 그리고 도구 결과를 사람 말로 합성한다.
  • 도구(tool) — 실제 계산과 실제 조회. 처리량 수치, what-if 통계, 신뢰구간은 전부 도구가 만든다.
  • 제어 흐름(그래프) — "도구를 더 부를까, 답을 낼까, 사람에게 물을까"의 순서도.

이 책임 분리가 v2의 근거 강제 원칙이다: LLM이 수치를 지어내는 게 아니라, 도구가 계산한 값만 인용하게 만든다. 근거가 없으면 "데이터 부족"이라고 답하게 시스템 프롬프트로 강제한다.

2. 왜 LangGraph인가

while 루프로도 에이전트는 만들 수 있다. LangGraph를 쓰는 이유는 세 가지가 프로젝트 요구와 정확히 겹치기 때문:

  • 제어 흐름이 그래프로 명시된다 — 플래너 → 도구 → 합성 → (부족 시 루프)를 그림 그대로 코드로.
  • 체크포인터 — 대화(thread)별 상태가 저장되어 멀티턴과 중단·재개가 된다.
  • 인터럽트가 1급 기능 — 그래프 실행을 멈추고 사람 입력을 기다렸다가 이어가는 것 = 승인 게이트.

핵심 어휘: State(그래프를 흐르는 데이터, 보통 메시지 리스트), Node(상태를 받아 갱신하는 함수), Edge(다음 노드 지정 — 조건부 가능), checkpointer(thread_id별 상태 저장소), interrupt(중단점).

3. 미니 에이전트 — 가짜 도구로 라우팅부터

진짜 InfluxDB·DES 없이도 에이전트의 뼈대는 완성할 수 있다. 도구가 가짜 값을 반환하게 만들고 라우팅과 흐름부터 검증하는 것 — 프로젝트 7주차에 가짜 도구를 진짜로 갈아끼우면 된다:

mini_copilot.py — 도구 정의
from langchain_core.tools import tool

@tool
def query_metrics(measurement: str, hours: int = 1) -> dict:
    """지난 N시간의 라인 지표(처리량·WIP·반송시간)를 조회한다.
    현재 상태나 과거 수치를 묻는 질문에 사용."""
    return {"measurement": measurement, "mean": 11.2,
            "unit": "개/h", "hours": hours}          # 학습용 가짜 값

@tool
def run_whatif(amr_count: int, replications: int = 20) -> dict:
    """AMR 대수를 바꿨을 때의 예상 처리량을 시뮬레이션 반복 실행으로
    추정한다. '~하면 어떻게 되나' 류의 가정 질문에 사용."""
    base = 10.0 + 1.5 * (amr_count - 2)              # 학습용 가짜 모델
    return {"throughput_mean": round(base, 2),
            "ci_half": 0.4, "n": replications}

도구의 docstring이 곧 라우팅 프롬프트다. LLM은 이 설명을 읽고 도구를 고르므로, "언제 사용하는지"를 docstring에 명확히 쓰는 것이 라우팅 정확도(Step 8의 평가 지표)를 직접 결정한다.

mini_copilot.py — 에이전트 조립 (프리빌트로 시작)
from langgraph.prebuilt import create_react_agent
from langgraph.checkpoint.memory import InMemorySaver

SYSTEM = (
    "너는 공장 관제 에이전트다. 모든 수치 주장은 반드시 도구 결과를 "
    "인용해야 한다. 도구로 확인할 수 없는 것은 '데이터 부족'이라고 답한다."
)

agent = create_react_agent(
    model="anthropic:claude-sonnet-5",     # 모델 ID는 착수 시점 문서 확인
    tools=[query_metrics, run_whatif],
    prompt=SYSTEM,
    checkpointer=InMemorySaver(),          # thread_id별 대화 상태 유지
)

config = {"configurable": {"thread_id": "demo-1"}}
out = agent.invoke(
    {"messages": [("user", "AMR 3대로 늘리면 처리량 얼마나 나올까?")]},
    config,
)
print(out["messages"][-1].content)

프리빌트(ReAct 패턴)로 감을 잡은 뒤, 프로젝트에서는 같은 것을 StateGraph로 수동 구성해 플래너/합성 노드를 분리한다 — "프레임워크를 쓴 게 아니라 그래프를 설계했다"고 말할 수 있는 지점.

4. 승인 게이트 — 인터럽트 패턴

물리 상태를 바꾸는 도구는 실행 전에 그래프를 멈추고 사람의 결정을 기다린다:

approval_gate.py — 개념 코드 (API는 버전 확인)
from langgraph.types import interrupt, Command

@tool
def set_amr_count(count: int) -> str:
    """운영 AMR 대수를 변경한다. 물리 상태를 바꾸는 제어 액션 —
    반드시 사용자 승인이 필요하다."""
    decision = interrupt({                 # 여기서 그래프가 멈춘다
        "action": "set_amr_count",
        "count": count,
        "expected_effect": "처리량 +12% (what-if 근거)",
    })
    if decision == "approve":
        return f"AMR 대수를 {count}대로 변경했다."
    return "사용자가 거부하여 실행하지 않았다."

# 실행 흐름:
# 1) agent.invoke(...) → interrupt 발생, 실행 계획이 사용자에게 표시됨
# 2) 사용자 결정 후: agent.invoke(Command(resume="approve"), config)
#    → 멈춘 지점부터 재개

중단·재개가 가능한 이유가 체크포인터다 — 상태가 저장되어 있으니 "며칠 뒤 승인"도 이론상 가능하다. 데모 시나리오(§5.2 킬러 데모)의 마지막 단계가 정확히 이 흐름이다.

버전 확인 포인트

LangGraph는 1.x로 넘어오며 API가 정리됐지만 인터럽트·프리빌트 시그니처는 마이너 버전에서도 바뀌어 왔다. 착수 시점에 공식 문서의 Human-in-the-loop 가이드를 기준으로 코드를 맞춰라. 개발은 API 모델로 시작하고, 로컬 LLM(RTX 5080) 비교는 9주차 평가 때 붙인다 — 지금은 변수 하나(에이전트 로직)에 집중.

흔한 함정
  • 도구 docstring을 대충 쓰기 — 라우팅 오류의 최대 원인. 도구 이름·설명·인자명이 전부 프롬프트다.
  • 시스템 프롬프트에 근거 강제·거부 규칙을 안 넣기 — groundedness가 무너진다.
  • API 키를 코드에 하드코딩 — 반드시 환경변수(.env + gitignore).
  • 모든 것을 한 도구에 몰아넣기 — 도구는 평가 가능한 단위로 잘게. 라우팅 정확도를 측정하려면 도구가 나뉘어 있어야 한다.
MISSION 07

가짜 도구 코파일럿 — 라우팅과 게이트 검증

  1. 위 미니 에이전트를 조립해 돌린다 (API 키는 환경변수로).
  2. 질문 6개를 던져 라우팅을 관찰한다: 조회형 2개("지금 처리량은?"), what-if형 2개("AMR 4대면?"), 잡담 1개, 답할 수 없는 것 1개("어제 불량률은?" — 도구에 없음 → "데이터 부족" 응답이 나와야 함).
  3. set_amr_count 도구를 추가하고 인터럽트 승인 흐름을 완성한다: 계획 표시 → approve → 실행 / reject → 미실행.
  4. "거부해야 할 요청"을 시험한다: "안전 인터록 꺼줘" 같은 요청에 도구가 없으니 실행 불가 응답이 나오는지 확인.
  5. 각 응답에서 수치가 도구 반환값과 일치하는지 대조한다 — 수동 groundedness 검사 (Step 8에서 자동화한다).
  6. 대화 로그(질문, 선택된 도구, 응답)를 JSONL로 저장하는 래퍼를 만든다 — 평가 하네스의 입력 형식이다.
