01

SimPy 이산사건 시뮬레이션 — DES 코어의 원형 만들기

코딩 ~10h 프로젝트 1–2주차 직결

학습 목표

  • 이산사건 시뮬레이션이 고정 타임스텝 시뮬레이션과 어떻게 다른지 설명할 수 있다.
  • SimPy의 세 기둥 — Environment, 프로세스(제너레이터), 자원(Resource/Store) — 을 사용할 수 있다.
  • 스테이션 2개 + 버퍼로 구성된 미니 라인을 만들고 처리량·체류시간을 측정할 수 있다.
  • 고장(MTBF/MTTR) 모델을 추가하고, 시드 고정으로 재현 가능한 실행을 만들 수 있다.
프로젝트와의 연결 — v2 §4.2(a)

이 스텝의 미션 산출물이 곧 DES 코어의 원형이다. 프로젝트 1–2주차는 이 미니 라인을 스테이션 3~4개로 확장하고 what-if 실행기 API로 감싸는 작업이라, 여기서 만든 코드를 그대로 가져간다.

1. 이산사건 패러다임 — 시간은 점프한다

게임 엔진이나 물리 시뮬레이터는 시간을 잘게 쪼개(예: 1/60초) 매 스텝 세계 전체를 갱신한다. 이산사건 시뮬레이션(DES)은 반대다. "다음에 무슨 일이 언제 일어나는가"만 이벤트 달력에 쌓고, 시계를 다음 이벤트 시각으로 점프시킨다. 가공 완료, 부품 도착, 고장 발생처럼 상태가 바뀌는 순간 사이의 시간은 계산하지 않는다.

  • 그래서 빠르다 — 8시간 조업 시뮬레이션이 수 밀리초에 끝난다. what-if 실행기가 수백 회 반복을 감당하는 근거.
  • 그래서 로직에 강하다 — 대기, 점유, 방출 같은 흐름 제약을 자연스럽게 표현한다.
  • 대신 물리는 없다 — AMR의 실제 주행 거동은 Isaac의 몫이다 (역할 분리, Step 0).

2. SimPy의 세 기둥

SimPy에서 세상은 Environment(시계+이벤트 달력) 위에서 도는 프로세스들의 집합이다. 프로세스는 평범한 파이썬 제너레이터 함수이고, yield가 "이벤트가 일어날 때까지 나는 잔다"를 뜻한다.

hello_des.py — 최소 동작 예제
import simpy

def machine(env, name, cycle_time):
    while True:
        yield env.timeout(cycle_time)      # cycle_time만큼 잔다
        print(f"[{env.now:6.1f}] {name} 1개 가공 완료")

env = simpy.Environment()
env.process(machine(env, "ST1", 5.0))      # 프로세스 등록
env.run(until=20)                          # 시뮬레이션 시각 20까지 실행

자원은 두 종류만 알면 시작할 수 있다:

  • Resource(env, capacity=n) — 동시에 n개까지 점유 가능한 것 (설비, 작업자). with res.request() as req: yield req 패턴으로 점유한다.
  • Store(env, capacity=n) — 물건이 쌓이는 곳 (버퍼, 대기열). put은 가득 차면, get은 비어 있으면 자동으로 블로킹된다.

3. 미니 생산 라인 — 버퍼 블로킹이 곧 현실

mini_line.py — 소스 → ST1 → 버퍼 → ST2
import simpy
import random

random.seed(42)

def part_source(env, out_buf, mean_interval):
    i = 0
    while True:
        yield env.timeout(random.expovariate(1 / mean_interval))
        i += 1
        yield out_buf.put(f"part-{i}")       # 버퍼가 가득이면 여기서 대기

def station(env, name, in_buf, out_buf, ct_mean, done_log):
    while True:
        part = yield in_buf.get()            # 입력이 올 때까지 대기
        ct = random.triangular(ct_mean * 0.8, ct_mean * 1.2, ct_mean)
        yield env.timeout(ct)                # 가공
        if out_buf is not None:
            yield out_buf.put(part)          # 다음 버퍼로 (가득이면 블로킹)
        else:
            done_log.append((env.now, part)) # 완성품 기록

env = simpy.Environment()
buf1 = simpy.Store(env, capacity=3)
buf2 = simpy.Store(env, capacity=3)
done = []

env.process(part_source(env, buf1, mean_interval=6.0))
env.process(station(env, "ST1", buf1, buf2, ct_mean=5.0, done_log=done))
env.process(station(env, "ST2", buf2, None, ct_mean=5.5, done_log=done))
env.run(until=8 * 60)                        # 8시간 조업 (분 단위)

print(f"8시간 생산량: {len(done)}개")
print(f"시간당 처리량: {len(done) / 8:.1f}개/h")

주목할 것은 블로킹이 공짜로 현실을 재현한다는 점이다. ST2가 느리면 buf2가 차고, buf2가 차면 ST1이 put에서 멈추고(막힘, blocking), 결국 상류 전체가 정체된다 — 실제 라인의 병목 전파와 같은 메커니즘이다. 프로젝트에서는 "완성품을 다음 스테이션으로 옮기는" 자리에 반송 오더 발행이 들어간다.

4. 고장 모델 — MTBF와 MTTR

설비는 평균 MTBF 간격으로 고장 나고, 평균 MTTR 동안 수리된다. 둘 다 지수분포로 두는 것이 관례적 출발점이다. 가장 단순한 구현은 스테이션이 참조하는 가용 플래그를 두는 방식:

breakdown.py — 가용성 게이트 방식
def breakdown_process(env, state, mtbf, mttr):
    while True:
        yield env.timeout(random.expovariate(1 / mtbf))
        state["up"] = False                  # 고장
        yield env.timeout(random.expovariate(1 / mttr))
        state["up"] = True                   # 수리 완료

# station 루프의 가공 직전에 삽입:
#   while not state["up"]:
#       yield env.timeout(0.5)              # 수리 대기
더 깔끔한 방법

폴링 대신 SimPy 이벤트나 PreemptiveResource(가공 중 고장이 작업을 중단시키는 선점 방식)를 쓰는 게 정석이다. SimPy 공식 문서의 Machine Shop 예제가 선점 방식의 표준 구현이니, 미션을 끝낸 뒤 꼭 비교해서 읽어라.

5. 재현성 — 시드와 난수 스트림 분리

what-if 실험은 "파라미터만 바꿨을 때"의 차이를 봐야 한다. 모든 난수를 시드 하나에서 뽑으면 파라미터 변경이 난수 소비 순서를 흐트러뜨려 비교가 오염된다. 용도별로 난수 스트림을 분리하는 습관을 처음부터 들여라:

rng_streams.py
import numpy as np

rng_arrival = np.random.default_rng(101)   # 부품 도착 전용
rng_process = np.random.default_rng(202)   # 가공 시간 전용
rng_failure = np.random.default_rng(303)   # 고장 전용

interarrival = rng_arrival.exponential(6.0)
cycle = rng_process.triangular(4.0, 5.0, 6.0)
흔한 함정
  • yield를 빠뜨리면 시간이 소비되지 않고 그냥 지나간다 — 에러도 안 나서 제일 위험하다.
  • 시간 단위(초/분)를 파일마다 다르게 쓰면 나중에 지옥이 된다. 분 단위로 통일하고 상수로 명시하라.
  • 시뮬레이션 초반(빈 라인에서 채워지는 구간)은 통계를 오염시킨다 — warm-up 구간 제거는 Step 2에서 다룬다.
  • print로 관측하지 말고 (시각, 이벤트) 튜플을 리스트에 쌓아라. 그 리스트가 Step 3에서 InfluxDB로 들어간다.
