IM-Twin 사전 학습 Phase C · 3주차 직전

05

Isaac Sim 입문 — 트윈에 몸을 입히기

3D+통합 ~14h 최대 리스크 구간 — 시간 박스 필수

학습 목표

  • Omniverse/USD의 핵심 개념(스테이지, prim, 레이어)을 이해한다.
  • Isaac Sim을 설치하고 prebuilt 자산(창고, AMR)으로 씬을 구성한다.
  • ROS2 브리지를 활성화해 시뮬레이션 속 로봇을 토픽으로 움직인다.
  • 강등 게이트(→ Gazebo)의 판단 기준과 전환 비용을 이해한다.
프로젝트와의 연결 — v2 §4.2(b), §6 게이트

디지털트윈 직무에서 3D 데모 영상은 가장 강한 한 방이고, Isaac Sim(Omniverse)은 그 자체가 채용 키워드다. 동시에 v2 기획서가 지정한 최대 리스크이기도 하다. 이 스텝의 목표는 마스터가 아니라 "오더를 받으면 움직이는 로봇이 있는 창고 씬" — 그 이상은 프로젝트 3–5주차의 몫이다.

1. Omniverse와 USD — 세계관 먼저

Isaac Sim은 NVIDIA Omniverse 플랫폼 위에 올라간 로봇 시뮬레이터다. 밑바닥에 깔린 개념이 USD(Universal Scene Description, 픽사가 만든 3D 장면 기술 체계)라서, 어휘 몇 개만 잡으면 UI 전체가 읽힌다:

  • 스테이지(Stage) — 지금 열려 있는 장면 전체. 하나의 USD 파일 트리다.
  • 프림(Prim) — 스테이지 트리의 모든 노드 (로봇, 바닥, 조명, 카메라). 각 prim은 속성(위치, 물리, 재질)을 가진다.
  • 레이어 합성 — 원본을 건드리지 않고 위에 겹쳐서 수정하는 비파괴 편집. 참조(Reference)로 기존 자산을 씬에 "링크"해 쓴다 — 우리가 창고와 로봇을 가져다 쓰는 메커니즘.
  • 물리(PhysX) — Play 버튼을 눌러야 물리가 돈다. "로봇이 안 움직여요"의 절반은 Play를 안 누른 것.

면접 어휘로: "USD 기반 씬 그래프 위에서 prebuilt 자산을 참조로 합성해 씬을 구성했다"까지 말할 수 있으면 충분하다.

2. 설치 — 하루를 배정하라

  • 요구사항: RTX GPU 필수 (RTX 5080 충분), 최신 NVIDIA 드라이버, 디스크 수십 GB, RAM 32GB 권장.
  • 첫 실행은 원래 느리다 — 셰이더 컴파일에 수 분에서 수십 분. 멈춘 게 아니다.
  • 설치 검증: 빈 스테이지에 큐브 하나 추가 → Play → 중력으로 떨어지면 물리까지 정상.
버전 확인 포인트 (중요)

Isaac Sim은 오픈소스화(2025) 이후 배포 방식·확장(Extension) 이름이 버전마다 달라져 왔다. 설치 방법(pip/바이너리/컨테이너), 지원 ROS2 배포판, RTX 50 시리즈 드라이버 호환은 반드시 착수 시점의 공식 문서 기준으로 확인하라. 이 페이지는 개념과 작업 순서만 제공한다.

3. 첫 씬 — 만들지 말고 가져와라

v2 기획서의 원칙: 직접 모델링 최소화, prebuilt 자산 최대 활용. 작업 순서:

  1. 자산 브라우저(Isaac Assets)에서 창고(warehouse) 환경을 검색해 스테이지에 로드한다.
  2. 로봇 자산에서 AMR을 추가한다 — Nova Carter 계열이 레퍼런스 AMR 자산이다 (버전에 따라 명칭·구성 상이).
  3. Stage 트리에서 로봇 prim을 선택해 위치를 옮겨보고, Property 패널에서 속성 구조를 훑어본다.
  4. Play를 누르고 물리가 도는 상태에서 로봇이 바닥에 안착해 있는지 확인한다.

여기서 씬을 꾸미고 싶은 유혹이 온다. 참아라. 씬 1개 원칙(§9)은 이 유혹 때문에 존재한다.

4. Python 스크립팅 — 두 가지 모드

  • GUI Script Editor — 실행 중인 씬에 파이썬을 바로 쏘는 REPL. prim 조작, 위치 읽기 같은 실험은 전부 여기서. 학습 단계의 주력.
  • Standalone/Headless — 스크립트가 시뮬레이터를 직접 기동·제어. 재현 가능한 데모와 자동화용. 프로젝트 후반(데모 영상 뽑기)에 필요해진다.

5. ROS2 브리지 — 두 세계를 잇는 순간

Isaac Sim의 ROS2 Bridge 확장을 활성화하면 시뮬레이션 안의 로봇·센서가 ROS2 토픽으로 노출된다. 확인 절차는 Step 4에서 배운 도구 그대로:

terminal — 브리지 검증
# Isaac Sim 실행 + ROS2 Bridge 확장 활성화 + Play 상태에서:
ros2 topic list                    # /cmd_vel, /odom 계열이 보여야 함
ros2 topic echo /odom              # 로봇 위치가 흐르는지 확인

# 키보드로 로봇 몰아보기 (조향 데모의 표준 도구):
ros2 run teleop_twist_keyboard teleop_twist_keyboard

토픽이 안 보이면 순서대로 의심하라: ① Play 상태인가 ② 브리지 확장이 켜져 있나 ③ Isaac과 터미널의 ROS2 배포판이 일치하나 ④ ROS_DOMAIN_ID가 같나 ⑤ QoS 불일치(Step 4) 아닌가.

6. 강등 게이트 — 자존심이 아니라 설계다

v2 로드맵의 게이트: 프로젝트 4주차 말까지 "오더 수신 → AMR A→B 반송 → 완료 회신"이 안 되면 Gazebo로 전환. 전환 비용이 작은 이유는 아키텍처에 있다 — DES와 시뮬레이터는 토픽 계약(/transport/order, /transport/done)으로만 묶여 있어서, Isaac을 빼고 Gazebo(+Nav2 튜토리얼 스택)를 꽂아도 브리지·DES·파이프라인·에이전트는 전부 그대로다. Gazebo Harmonic + Nav2는 문서와 커뮤니티 자료의 왕도가 있으니, 게이트 발동은 실패가 아니라 리스크 관리가 작동한 것이다.

흔한 함정
  • Play 없이 "왜 안 움직이지" 30분 — 물리는 Play 상태에서만 돈다.
  • 커스텀 로봇/커스텀 창고 모델링 시작 — 즉시 중단하라. prebuilt가 원칙이다.
  • 렌더 품질 욕심으로 프레임 저하 — 학습·개발 중엔 렌더 세팅을 낮춰라. 예쁜 화면은 데모 영상 찍을 때만.
  • 드라이버/버전 불일치로 크래시 — 버전 조합(드라이버·Isaac·ROS2)을 기록해두고 한 번에 하나만 바꿔라.
  • 시간 박스 위반 — 이 스텝 전체가 안 풀려도 Gazebo 경로가 있다. 매몰비용에 끌려가지 마라.
MISSION 05

움직이는 창고 — 브리지 원형에 진짜 로봇 꽂기

  1. Isaac Sim을 설치하고 큐브 낙하로 물리를 검증한다 (버전 조합 기록).
  2. 창고 환경 + AMR 자산으로 씬을 구성하고 저장한다.
  3. ROS2 브리지를 켜고 teleop_twist_keyboard로 로봇을 몰아본다.
  4. Script Editor에서 로봇 prim의 위치를 읽고 출력해본다 — "시뮬레이션 상태를 코드로 읽는" 첫 경험.
  5. (도전) Step 4의 amr_mock을 대체하는 최소 노드를 만든다: /transport/order 수신 시 /cmd_vel로 로봇을 몇 초 전진시키고 /transport/done을 발행. 내비게이션 없이 직진이면 충분하다 — 루프가 닫히는 게 목적.
  6. 주행 장면을 30초 영상/GIF로 남긴다 — 첫 데모 자산.
