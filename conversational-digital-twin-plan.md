# 대화형 디지털 트윈 (Conversational Digital Twin) 프로젝트 기획서

> 산업 표준을 준수하는 가상 공장 디지털 트윈을 구축하고, 그 위에 자연어 인터페이스(LLM 에이전트)를 얹는 1인 포트폴리오 프로젝트.

**작성일**: 2026-07-02 · **기간**: 10주 · **인원**: 1인 · **하드웨어 지출**: 0원

---

## 1. 한 문장 정의

"OPC UA / ISA-95 / Sparkplug B 산업 표준으로 모델링된 가상 생산 라인 디지털 트윈을 만들고,
현장 엔지니어가 자연어로 조회·진단·제어(승인 기반)할 수 있는 팩토리 코파일럿을 그 위에 구축한다."

## 2. 왜 이 프로젝트인가

- **트렌드 정합성**: 실물 투자 전 디지털 트윈에서 검증하는 Simulate-then-Procure 방식이 업계 표준 워크플로우로 자리잡음. 제조업체의 LLM 관심도 급등(언어 기반 진단·코파일럿 수요).
- **직무 정합성**: 디지털 트윈/공정 시뮬레이션 엔지니어(제품 1) + 제조 AI/스마트팩토리 SW(제품 2) 두 직군에 동시 대응.
- **차별화**: 기계공학 도메인 지식(회전기계 물리, 결함 주파수, 생산 시스템 이론)이 시뮬레이터와 에이전트 진단 로직 양쪽에 코드로 증명됨.
- **희소성**: "에이전트 + 제조 도메인 + 산업 표준" 조합의 취준생 포트폴리오는 드묾.

## 3. 구조: 2제품, 2저장소

무게중심이 두 개인 문제를 저장소 분리로 해결한다. 어느 시점에 멈춰도 완결된 결과물이 남는다.

| | 제품 1: `virtual-factory-twin` | 제품 2: `factory-copilot` |
|---|---|---|
| 정체성 | 산업 표준 기반 가상 공장 오픈소스 | 트윈을 소비하는 자연어 관제 에이전트 |
| 타겟 직무 | 디지털 트윈, 공정 시뮬레이션, 생산기술 | 제조 AI, 스마트팩토리 SW |
| 단독 완결성 | OPC UA 가상 공장 + 검증 리포트로 완결 | 트윈 저장소를 의존성으로 사용 |

## 4. 제품 1 — virtual-factory-twin

### 4.1 아키텍처

```
[SimPy 이산사건 라인]  ←  [물리 신호 생성기 (진동·전류·온도)]
        │
        ├─→ [OPC UA 서버]  ← ISA-95 자산 계층 (Site/Area/Line/Cell/Equipment)
        │
        └─→ [MQTT (Sparkplug B 페이로드)] → [InfluxDB] → [Grafana]
```

### 4.2 핵심 구성요소

**(a) 이산사건 라인 시뮬레이터 (SimPy)**
- 스테이션 3~4개 + 버퍼 + 반송 구간으로 구성된 직렬 라인.
- 확률적 사이클타임, 랜덤 고장/수리(MTBF·MTTR), 시나리오 스크립트(정상 / 점진 열화 / 급성 고장) 재현 가능.

**(b) 물리 기반 신호 생성기**
- 설비별 회전 주파수 하모닉스 기반 진동 파형 생성.
- 열화 진행 시 베어링 결함 주파수(BPFO / BPFI / BSF / FTF — 기하 파라미터와 RPM으로 계산) 대역 에너지가 물리적으로 올바르게 성장.
- 전류·온도 채널 동반 생성(부하-전류 상관 반영).
- 이 모듈 자체가 기계공학 도메인 지식의 1차 증명.

**(c) 산업 표준 계층 (축 1 — 채용 가독성 핵심)**
- **OPC UA 서버**: 각 가상 설비의 상태·공정변수를 OPC UA 노드로 노출 (파이썬 asyncua 계열 구현체 사용, 셋업 시점에 최신 라이브러리 확인).
- **ISA-95 자산 모델**: Enterprise → Site → Area → Line → Cell → Equipment 계층으로 주소 공간 설계.
- **Sparkplug B**: MQTT 페이로드를 산업 규격(NBIRTH/NDATA 등 토픽 네임스페이스 포함)으로 발행.
- 효과: "SimPy를 써봤다" → "제조 현장 데이터 아키텍처를 이해한다"로 격상.

**(d) 시뮬레이션 정합성 검증 (축 2 — 공학적 신뢰성 핵심)**
- Little의 법칙(L = λW) 검증: 시뮬레이션 실측 WIP·처리율·체류시간의 일치 확인.
- 대기행렬 이론 예측치와 버퍼 점유율 비교.
- 버퍼 크기·고장률 파라미터 스윕에 따른 처리량 민감도 분석 → 병목 식별 리포트.
- 효과: "시뮬레이터가 옳다는 걸 어떻게 아는가?"라는 면접 질문에 대한 정면 답변.

### 4.3 산출물
- Docker Compose 원커맨드 기동 (시뮬레이터 + OPC UA + MQTT 브로커 + InfluxDB + Grafana).
- Grafana 실시간 대시보드 (라인 처리량, 설비 상태, 진동 스펙트럼).
- 검증 리포트 (Little의 법칙 검증 그래프, 병목·민감도 분석) — `docs/validation.md`.
- 데모 영상 (정상 → 열화 → 고장 시나리오 재생).

## 5. 제품 2 — factory-copilot

### 5.1 에이전트 그래프 (LangGraph)

```
사용자 질의 → [플래너: 의도 분석·도구 선택]
                ├─ 시계열 조회 (Text-to-Flux, InfluxDB)
                ├─ 이상 진단 (결함 주파수 계산·스펙트럼 해석 내장)
                ├─ 문서 검색 (설비 매뉴얼 RAG)
                └─ 설비 제어 (OPC UA write, 승인 게이트 필수)
              → [합성·검증: 근거 인용·응답 생성] → (부족 시 플래너로 루프)
```

### 5.2 설계 원칙
- **승인 게이트(Human-in-the-loop)**: 물리 상태를 바꾸는 액션은 실행 계획 제시 → 사용자 승인 → 실행. LangGraph 인터럽트로 구현. 제조 현장 안전 서사의 핵심.
- **환각 억제**: 진단은 LLM 추측이 아니라 도구가 계산한 결함 주파수·스펙트럼 수치를 근거로만 답변. 근거 없는 주장 시 "데이터 부족" 응답.
- **표준 경유 제어**: 코파일럿이 InfluxDB 직접 조작이 아닌 OPC UA를 통해 트윈을 읽고 제어 → 실공장 이식 가능한 구조임을 어필.
- **멀티턴 루프**: 단일 도구로 부족하면 스스로 추가 조회 결정 (multi-hop).

### 5.3 평가 (정량화 필수)
- 질의 30~50개 평가셋 구축 (조회/진단/문서/제어/거부해야 할 질의 혼합).
- 채점 항목: 도구 라우팅 정확도, 쿼리 정확성, 근거 충실성(groundedness), 승인 게이트 준수율.
- **API vs 로컬 LLM 비교**: 클라우드 API 모델 vs RTX 5080 로컬 서빙 오픈소스 모델(Qwen/Llama 계열, 시점 기준 최신 확인)을 동일 평가셋으로 측정 → 정확도 격차, 응답 지연, 폐쇄망 요구 충족 최소 모델 크기 도출. 온프레미스 제약이 있는 제조업 특성에 정면 대응하는 차별화 파트.

### 5.4 산출물
- 채팅 UI (Streamlit 등 경량) + 데모 영상 (조회 → 진단 → 매뉴얼 → 승인 제어 시나리오).
- 평가 리포트 (`docs/evaluation.md`): 라우팅 정확도, groundedness, API vs 로컬 비교표.

각 2주 구간은 단독 완결 원칙을 따른다. 어디서 멈춰도 포트폴리오가 남는다.

## 7. 기술 스택

| 계층 | 기술 | 선택 이유 (면접 답변 겸용) |
|---|---|---|
| 시뮬레이션 | Python, SimPy, NumPy/SciPy | 이산사건 표준 라이브러리, 물리 신호는 직접 구현이 증명력 높음 |
| 산업 프로토콜 | OPC UA(asyncua 계열), MQTT + Sparkplug B | 실공장 데이터 아키텍처 표준. HTTP 대비 텔레메트리 적합성 실측으로 입증 |
| 저장·가시화 | InfluxDB, Grafana, Docker Compose | 시계열 특화 + 가시화 코딩 생략 → 1인 리소스를 핵심에 집중 |
| 이상 진단 | FFT 피처 + 결함 주파수 규칙(+경량 오토인코더 선택) | 진단 근거의 물리적 설명 가능성 우선 |
| 에이전트 | LangGraph, (API LLM + 로컬 LLM 병행) | 그래프 기반 제어 흐름·인터럽트 지원, 폐쇄망 대응 비교 실험 |
| UI | Streamlit | 1인 프로젝트 UI 비용 최소화 |

## 8. 실측 지표 체크리스트 (전부 본인 시스템에서 측정)

- [ ] 종단 간 수집 지연 (신호 생성 → InfluxDB 기록, ms) — 조건 명시(페이로드 크기, 발행 주기)
- [ ] 초당 처리 데이터 포인트, 유실률
- [ ] Little의 법칙 검증 오차율, 병목 스테이션 식별 결과
- [ ] 결함 시나리오별 탐지 리드타임 (열화 시작 → 경보까지)
- [ ] 코파일럿 도구 라우팅 정확도, groundedness 점수, 승인 게이트 준수율
- [ ] API vs 로컬 LLM: 정확도 격차(%p), 평균 응답 지연(s), 로컬 최소 가용 모델 크기

주의: 외부 자료의 예시 수치를 절대 차용하지 않는다. 모든 숫자는 측정 조건과 함께 기록한다.

## 10. 면접 방어 포인트 (예상 질문 → 준비된 답)

1. "시뮬레이션 데이터인데 신뢰할 수 있나?" → 물리 법칙 기반 신호 설계 + Little의 법칙 정합성 검증 + (선택) 공개 데이터셋 교차 검증.
2. "왜 MQTT/OPC UA인가?" → 프로토콜별 역할 구분(텔레메트리 vs 정보 모델·제어)과 실측 지연 수치로 답변.
3. "LLM이 오작동하면?" → 승인 게이트, 도구 계산 근거 강제, 거부 케이스 포함 평가셋.
4. "폐쇄망 공장에서 되나?" → 로컬 LLM 비교 실험 결과로 답변.
5. "기계공학 전공이 어디에 쓰였나?" → 결함 주파수 신호 생성기, 스펙트럼 진단 도구, 생산 시스템 이론 검증.
